## 1. 프로젝트 개요
시스템의 결합도를 낮추고 유지보수성을 극대화하는 아키텍처 설계를 지향하며 구축한 소규모 그룹웨어 시스템입니다. 프론트엔드와 백엔드의 리포지토리를 철저히 분리하고 도메인 간 독립성을 부여하여, 향후 서비스 확장 및 유지보수 시 유연하게 대응할 수 있는 기반을 마련했습니다. 실무 프로젝트 경험을 통해 단순한 기능 구현을 넘어, 서비스의 초기 기반을 기획하고 팀의 협업 인프라를 주도적으로 구축했습니다.

* **진행 기간**: 2025.10.22 ~ 2025.11.26 (5주)
* **개발 인원**: 5명
* **담당 역할**: 프로젝트 리더로서 팀 개발 표준 수립 및 릴리즈 관리를 수행했으며, 백엔드 아키텍처 설계와 서버 인프라(On-premises/Cloud) 구축 및 CI/CD 파이프라인 구성을 담당했습니다.
* **주요 대상**: 스타트업 및 중소기업
* **핵심 목적**: 비즈니스 운영에 필요한 인사, 전자결재 등 핵심 그룹웨어 기능을 도메인 기반으로 모듈화하고 서비스화했습니다.
* **리포지토리**: 
  * Backend: [https://github.com/khyeongtaek/startup_BE](https://github.com/khyeongtaek/startup_BE)
  * Frontend: [https://github.com/khyeongtaek/startup_FE](https://github.com/khyeongtaek/startup_FE)

## 5. 프로젝트 리더십 및 협업 문화
기능 구현에 앞서 팀의 개발 환경과 규칙을 체계화하여 프로젝트 리스크를 최소화했습니다. 특히 **한정된 리소스 내에서 개발 효율성을 극대화하기 위해 상용 템플릿 도입과 같은 전략적 의사결정을 수행했으며,** 이를 통해 확보한 리소스를 백엔드 핵심 도메인 로직과 안정적인 인프라 설계에 집중 투자했습니다.

* **Git 워크플로우 및 코드 리뷰 체계 확립**: `main` - `dev` - `feature` 브랜치 전략을 수립하여 관리했습니다. 개인 환경에서 테스트가 완료된 코드만 PR을 생성하도록 하고, 리뷰 및 병합 프로세스를 통제하여 `dev` 브랜치의 무결성을 엄격하게 유지했습니다.
* **표준화 가이드라인(`CONTRIBUTING.md`) 배포**: 프로젝트 초기, 네이밍 컨벤션 및 API 설계 기준의 혼선이 개발 지연으로 이어질 것을 예측했습니다. 이를 방지하기 위해 표준 가이드를 선제적으로 수립·배포하여 불필요한 커뮤니케이션 비용을 최소화했습니다. 덕분에 확보된 리소스를 개발에 재투자하여, 초기 기획 시 후순위로 밀렸던 세부 디테일까지 보완하며 프로젝트의 완성도를 한 단계 높일 수 있었습니다.
* **비용 효율적인 환경 분리 및 배포 자동화**: 클라우드 서비스의 무료 플랜 이용 시 발생할 수 있는 예기치 못한 과금 문제를 방지하기 위해, 데이터베이스(MySQL)는 개인 소유의 Synology NAS에 Docker를 활용하여 구축했습니다. 웹 서버(Nginx)와 애플리케이션 운영 서버는 AWS EC2에서 운영하며, GitHub Actions를 통해 CI/CD 파이프라인을 구축하여 안정적인 자동 배포 환경을 마련했습니다.

## 7. 핵심 구현 기능 및 트러블 슈팅

### 7.1. [코드 관리] 공통 코드 계층화 설계 및 시스템 확장성 확보
* **Situation**: 직급, 부서, 상태값 등 시스템 전반에 사용되는 코드형 데이터가 분산 관리될 경우, 운영진의 직접적인 제어가 어렵고 유지보수 효율이 저하되는 문제가 있었습니다.
* **Task**: 개발자의 개입 없이도 시스템 변수를 실시간으로 제어할 수 있는 중앙 집중형 공통 코드 관리 모듈을 구축하는 것이 목표였습니다.
* **Action**: 단일 테이블 내에서 부모 키를 통해 셀프 참조하여 대분류 소분류를 연결하는 관리 구조를 설계하고, 분산된 코드 조회 로직을 단일 API(`getCommonCodeByPrefixWithoutRoot`)로 통합하여 호출 방식을 개선했습니다.
* **Result**: 운영진이 직접 직급 등의 코드 추가나 비활성화 처리를 수행할 수 있게 되었으며, 저장 시점의 작성자 정보를 기록하여 데이터 변경의 명확한 추적성을 확보했습니다.
![코드관리](https://github.com/user-attachments/assets/e2bf9371-e0bf-4f71-a7e7-64c3a9741417)

### 7.2. [메뉴 관리] 동적 라우팅 시스템 구축을 통한 유지보수 공수 절감
* **Situation**: 메뉴 명칭이나 라우팅 정보가 소스 코드에 하드코딩될 경우, 구조 변경 시마다 프론트엔드 재빌드 및 배포가 필요한 비효율이 존재했습니다.
* **Task**: 배포 공정 없이 메뉴 구조를 유연하게 조정할 수 있는 동적 UI 렌더링 시스템을 구현해야 했습니다.
* **Action**: 메뉴 정보를 데이터화하고 Mapper 패턴을 적용해 DB 문자열 정보를 실제 컴포넌트로 변환했습니다. `MenuContext` 갱신 로직을 개선하여 코드 관리에서의 설정 변경이 사용자 화면에 즉시 반영되도록 구현했습니다.
* **Result**: 운영자가 메뉴의 활성 여부를 실시간으로 제어할 수 있게 되었으며, 메뉴 검색 기능을 통해 사용자가 원하는 경로로 즉시 이동할 수 있도록 편의성을 개선했습니다.
![메뉴관리](https://github.com/user-attachments/assets/c8285898-6dfa-49f3-a77b-a653e8e5a231)

### 7.3. [로그인] HttpOnly 쿠키와 Spring Security를 활용한 안전한 인증 프로세스
* **Situation**: 기존의 헤더 기반 JWT 인증은 브라우저 스토리지 보관 시 XSS 공격에 취약했으며, 토큰 만료 시 사용자가 잦은 재로그인을 수행해야 하는 UX 저하 문제가 있었습니다.
* **Task**: 보안 취약점을 근본적으로 차단하는 동시에, 토큰 재발급 로직을 통해 사용자가 중단 없이 서비스를 이용할 수 있는 안전한 인증 인프라를 설계해야 했습니다.
* **Action**: Spring Security를 연동하고, Access/Refresh Token을 클라이언트 스크립트에서 접근할 수 없는 `HttpOnly` 쿠키로 발급하도록 설계했습니다. 서버 측에서 Refresh Token의 유효성을 검증하고, 만료 시점에 맞춰 안전하게 재발급되는 로직을 구현했습니다.
* **Result**: 세션 탈취 위험을 근본적으로 차단하여 보안성을 강화했으며, 클라이언트 인터셉터 로직과 연계하여 사용자는 토큰 만료를 체감하지 않고 끊김 없는 서비스를 이용할 수 있게 되었습니다.
![로그인](https://github.com/user-attachments/assets/85609dd9-ef87-4135-bcfb-9240ad1db2db)

### 7.4. [인사관리] 대량 인사 데이터 연동 및 변경 이력 추적 기능 구현
* **Situation**: 프로젝트 특성상 다수의 사원 데이터를 한 번에 등록할 기능이 필요했으며, 데이터 변경 시 누가 언제 어떤 정보를 수정했는지 추적할 수단이 없어 관리자 기능의 완성도가 부족했습니다.
* **Task**: 대량의 데이터를 효율적으로 등록하는 수단을 마련하고, 인사 데이터의 생성 및 변경 이력을 추적 및 조회할 수 있도록 해야 했습니다.
* **Action**:
  * CSV 파일을 파싱하여 사원 데이터를 일괄 등록하는 기능을 구현하여 대량의 인사 데이터 연동을 자동화했습니다.
  * 신규 생성, 비밀번호 변경, 정보 수정 등 사원 데이터의 변경이 발생할 때마다 변경 시점과 내용을 기록하는 이력 테이블(`EmployeeHistory`)을 도입했습니다.
* **Result**: CSV 업로드를 통해 초기 데이터 세팅 및 인사 연동의 편의성을 높였으며, 인사 데이터의 모든 변경 과정을 체계적으로 추적할 수 있게 되어 시스템의 신뢰성을 향상시켰습니다.
![인사관리](https://github.com/user-attachments/assets/c844cae2-674c-4e86-a3b0-173f3b15a652)

### 7.5. [전자결재] JPA 최적화 및 동적 UI 분기 처리를 통한 결재 엔진 구현
* **Situation**: 다수의 결재선이 포함된 목록 조회 시 N+1 문제로 인한 성능 병목이 우려되었으며, 사용자의 상태에 따라 복잡한 UI 분기 처리가 필요했습니다.
* **Task**: 데이터 조회 성능을 개선하고 사용자 권한에 따른 정확한 결재 처리 환경을 제공하는 것이 목표였습니다.
* **Action**: JPA Fetch Join을 활용해 연관 엔티티를 단일 쿼리로 조회하도록 최적화했습니다. 접속자가 현재 결재 순번인지를 서버에서 자동 판단하여 권한이 있을 때만 승인/반려 버튼을 활성화하는 UI 로직을 적용했습니다.
* **Result**: 쿼리 실행 횟수를 최적화하여 DB 부하를 감소시켰으며, 승인 시 다음 결재자에게 즉시 알림을 발송하여 신속한 의사결정을 지원하도록 구현했습니다.

### 7.6. [마이페이지] 보안 검증 기반의 개인 정보 및 프로필 관리
* **Situation**: 중요 인사 데이터의 무결성을 보호하면서도 사용자의 개인화 정보를 안전하게 수정할 수 있는 권한 체계와 보안 절차가 요구되었습니다.
* **Task**: 보안이 중요한 정보 변경 시 검증을 거치도록 설계하고, 전역 UI에 프로필 이미지를 실시간 연동해야 했습니다.
* **Action**: 타인의 도용 방지를 위해 현재 비밀번호를 재검증한 후 비밀번호를 변경하도록 로직을 강화했습니다. 마이페이지에서 업로드한 프로필 이미지가 헤더, 인사기록 등 시스템 전반에 즉시 반영되도록 연동했습니다.
* **Result**: 안전한 정보 변경 프로세스를 확립했으며, 개인별 사번이나 부서 데이터는 보호하면서도 연락처나 사진 등 개인화 정보를 유연하게 관리할 수 있도록 구현했습니다.

## 8. 프로젝트 참여 소감
이번 프로젝트는 단순한 비즈니스 로직 구현을 넘어, '시스템 아키텍처의 유연성'과 '기반 코드의 무게감'을 깊이 체감한 계기였습니다.
초기에는 객체지향의 지향점인 '시스템 결합도를 낮추는 것'을 최우선 목표로 삼았습니다. 하지만 모든 기능의 뿌리가 되는 공통 코드와 인프라를 설계하며, 이러한 기반 모듈은 필연적으로 모든 도메인이 참조할 수밖에 없는 '중심점'이 됨을 경험했습니다. 모듈의 작은 변화가 전체 시스템에 미치는 파급력을 직접 확인한 후, 시스템 전역에 영향을 주는 이 필수적인 의존성을 어떻게 안전하게 관리할 것인가에 대해 많은 고민을 했습니다.
그 해결책으로 잦은 수정 없이도 다양한 데이터 타입을 수용할 수 있도록 데이터 모델의 범용성을 확보하는 데 집중했습니다. 이 과정을 통해 설계 단계에서 발생 가능한 다양한 상황을 미리 고려하는 것이 향후 유지보수 리스크를 낮추는 데 얼마나 핵심적인지 배울 수 있었습니다.
결과적으로 단순히 작동하는 코드를 작성하는 것을 넘어, "내가 만든 이 기반 코드가 팀원들의 개발 생산성과 전체 시스템의 안정성에 어떠한 영향을 미치는가"를 항상 먼저 생각하는 책임감을 갖추게 되었습니다. 이번 프로젝트에서의 고민과 경험을 바탕으로, 앞으로도 확장 가능하고 견고한 백엔드 생태계를 구축하는 데 기여하겠습니다.
